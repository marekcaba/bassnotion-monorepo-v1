================================================================================
1. HOW SCHEDULING WORKS
================================================================================

The scheduling system has 3 key layers:

A. EVENT COLLECTION (RegionScheduler.ts)
   ├─ Parse MIDI tick data
   ├─ Convert ticks → seconds: seconds = (ticks / 480) * (60 / BPM)
   ├─ Add region start time + countdown offset
   ├─ Round to 1ms precision for batching
   └─ Example: 960 ticks @ 120 BPM = 1.0 second

B. LOOK-AHEAD SCHEDULING (Scheduler.ts)
   ├─ Calculate window: scheduleUntil = currentTime + 150ms
   ├─ Filter events: only schedule if in [currentTime, currentTime+150ms]
   ├─ Sort: by priority (high first), then time
   ├─ Send to Tone.js for Web Audio execution
   └─ Update loop: every 20ms (50Hz polling)

C. AUDIO EXECUTION (Web Audio API)
   ├─ Tone.js schedules callback for exact audio time
   ├─ Uses sample-accurate timing when AudioWorklet available
   ├─ Precision: ±20 microseconds (1 sample @ 48kHz)
   └─ Hardware latency: ~10-50ms (consistent, not jitter)

================================================================================
2. FRAME-BASED TIMING (targetFrame/scheduleFrame)
================================================================================

Frame-based measurements use 48kHz audio sample rate:

1 frame = 1/48000 second = 20.83 microseconds

Frame conversions:
├─ 0.5 seconds   = 24,000 frames
├─ 1.0 second    = 48,000 frames
├─ 150ms (lookahead) = 7,200 frames
└─ 20ms (update)     = 960 frames

How they're used:
├─ scheduleFrame = frame count when event was scheduled
├─ targetFrame = frame count when event should play
├─ lookaheadMs = (targetFrame - scheduleFrame) / 48
└─ Example: (96000 - 88800) / 48 = 150ms lookahead ✓

================================================================================
3. VISUAL VS AUDIO TIMING
================================================================================

These are DIFFERENT systems:

AUDIO TIMING:
├─ Scheduled 150ms in advance (look-ahead)
├─ Executed by Web Audio API (not JavaScript!)
├─ Sample-accurate with AudioWorklet (< 1ms jitter)
├─ Based on AudioContext.currentTime
└─ No JavaScript garbage collection interference

VISUAL TIMING:
├─ Updated at 60fps (16.67ms per frame)
├─ JavaScript-based (can have GC jitter)
├─ Target sync accuracy: ±5ms from audio
├─ Drift correction: applied every 1000ms
└─ Good enough for UI (±5-20ms acceptable)

They serve different purposes - don't confuse them!

================================================================================
4. DOES 3% ACCURACY MAKE SENSE?
================================================================================

ANSWER: NO - Here's why:

3% Accuracy = 3% of note duration

Examples:
├─ 60 BPM quarter note:  1000ms → 3% = 30ms (bad!)
├─ 120 BPM quarter note: 500ms → 3% = 15ms (noticeable)
├─ 240 BPM 16th note:    125ms → 3% = 3.75ms (acceptable)

Problems:
├─ Too loose for slow tempos
├─ Changes with every tempo change
├─ Doesn't match professional audio standards
├─ Current system achieves MUCH better: < 5ms!

What BassNotion ACTUALLY achieves:
├─ ✓ Sample-accurate: < 1ms (with AudioWorklet)
├─ ✓ Cross-instrument sync: < 5ms (measured via diagnostic)
├─ ✓ Look-ahead: 150ms (prevents missed deadlines)
├─ ✓ Professional standard: Matches Pro Tools, Ableton, etc.

================================================================================
5. LOOK-AHEAD WINDOW (150ms)
================================================================================

Why 150ms?

Professional standard:
├─ Pro Tools, Ableton, Cubase: 50-200ms typical
├─ BassNotion: 150ms = middle ground
├─ Balances: stability vs latency

Safety margin:
├─ Reaction time for CPU jitter: 150ms
├─ Prevents missed deadlines: highly reliable
├─ Still low latency: < 5 frames @ 30fps

How it works:
├─ At t=1.85s: "Schedule all events in [1.85 - 2.0s]"
├─ At t=1.87s: "Schedule all events in [1.87 - 2.02s]"
├─ By t=2.0s:  Events already scheduled 150ms ago!
└─ Result: Rock-solid timing, no jitter

Update frequency:
├─ 20ms polling interval (50Hz)
├─ Checks for new events to schedule
├─ Ensures events don't miss deadline

7. SCHEDULING PIPELINE SUMMARY
================================================================================

1. Event Data
   ├─ MIDI ticks: 960 (2 beats @ 480 ticks/beat)
   ├─ Region start: 0.3 seconds
   └─ Time: 0.3 + (960/480)*(60/120) = 1.3 seconds

2. Scheduling Cycle
   ├─ At t=1.15s: Check [1.15 - 1.30s]
   ├─ Find: event at t=1.3s
   ├─ Schedule: Tone.Transport.schedule(callback, 1.3)
   └─ Look-ahead: 1.3 - 1.15 = 150ms ✓

3. Execution
   ├─ At t=1.3s (Web Audio API thread)
   ├─ Call: sampler.start(audioContext.currentTime)
   ├─ Timing: sample-accurate ±20µs
   └─ Audio output: ~10-50ms later (hardware latency)

4. Visual Feedback
   ├─ Next animation frame (≤16.67ms away)
   ├─ Read: audioContext.currentTime
   ├─ Render: UI update showing note
   └─ Sync error: typically ±5-10ms from audio

5. Diagnostic Measurement
   ├─ Record: scheduledTime, jsExecutionTime, frames
   ├─ Calculate: timing differences between instruments
   ├─ Report: max delta, trend analysis
   └─ Target: all instruments within 5ms

================================================================================
8. TEMPO COMPENSATION
================================================================================

When tempo changes (e.g., 120 BPM → 140 BPM):

Step 1: Snapshot beats at old tempo
├─ Current time: t=2.5s
├─ Beats accumulated: 2.5 * (120/60) = 5 beats
├─ Store: accumulatedBeats = 5

Step 2: Reset timing reference
├─ lastTempoChangeTime = 2.5s
├─ currentBPM = 140

Step 3: Calculate position with new tempo
├─ At t=2.6s: elapsed = 0.1s
├─ Beats since change: 0.1 * (140/60) = 0.233 beats
├─ Total: 5 + 0.233 = 5.233 beats
└─ Position: 5.233 / (140/60) = 2.244s (no jump!)

Result: Smooth transition, no position jumps

================================================================================
9. DRIFT PREDICTION & CORRECTION
================================================================================

How drift is detected:
├─ Kalman filter: tracks timing deviations
├─ Drift = actualTime - expectedTime
├─ Threshold: > 1ms is "significant"
└─ Trend: stable / increasing / decreasing

Predictive compensation:
├─ Velocity: rate of drift change
├─ Acceleration: rate of velocity change
├─ Predicts: future drift using physics model
└─ Allows: proactive correction before it matters

Visual sync correction:
├─ Measured every 1000ms
├─ Applied: drift offset += avgDrift * 0.1 (10% factor)
├─ Result: gradual sync recovery
└─ Never jerky or noticeable to user
