Recording-Latency Probe capture feasibility · docs/dev-tools

Answers the load-bearing question from docs/PRACTICE_TOOLS_FEASIBILITY.md → "Capturing the user's own bass recording." It measures the true acoustic round-trip of this machine + interface via a loopback click (app emits a click → it leaves the speaker/interface → the input picks it up → we measure the gap), then proves whether punch-in alignment can land a recorded take sample-accurate against a backing track. Nothing leaves the device. Use headphones-off / speaker-near-mic OR an interface loopback for the round-trip test; for the verdict that matters, plug in your real bass + interface.

1 · Audio input

1b · Live input meter — is your bass reaching the browser?

On a Focusrite Clarett you monitor your bass via Focusrite Control direct monitoring (analog, 0ms) — the browser does NOT play it back to your ears, and that's correct. This only checks the app is receiving the signal. Play your bass — the bar should jump. If it stays dead, the browser opened the wrong input (pick the Clarett's instrument channel in §1, not "Default"). For the round-trip test in §2, enable Focusrite Control's "Loopback" input and select it as the device, OR run a physical cable from an output to an input.

2 · Round-trip latency (loopback click)

Plays N short clicks out the output; listens on the input for each click's echo; reports the delay. This is the real number the browser path imposes on this rig — the "monitoring latency" that, if heard through the app, is the molasses/word-of-mouth killer. Run it with a loopback cable (interface out→in) for a clean read, or speaker-near-mic.

3 · Punch-in alignment test (does the OFFSET fix the take?)

The real question for a browser-only launch: even with latency, can we record then shift the take back by the measured offset so it sits in the pocket with the backing? This plays a metronome backing, records the input, auto-correlates the recorded clicks against the known backing grid, and reports the residual error after applying the measured offset. Tap your bass (or clap) on the beats during the count.

Thresholds: round-trip <12 ms = through-app monitoring feels live (browser fine); 12–25 ms = noticeable, hardware direct-monitoring recommended; >25 ms = through-app monitoring is molasses → daemon needed for that mode. Punch-in residual <3 ms (~144 samples @48k) = recorded takes align sample-accurate → record-then-review works browser-only regardless of monitoring latency.
